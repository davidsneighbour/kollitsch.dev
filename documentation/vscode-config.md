The workspace uses a three-file pipeline to keep VS Code settings reproducible without exposing machine-specific preferences in version control.

## Files

| File | Committed | Purpose |
| --- | --- | --- |
| `.vscode/settings.base.jsonc` | Yes | Shared project settings. Every collaborator gets these. |
| `.vscode/settings.local.jsonc` | Yes | Per-machine overrides (colors, personal preferences). Merges on top of base. |
| `.vscode/settings.json` | Yes (generated) | Final merged output. Written by the sync script. Do not edit manually. |

`settings.json` is the file VS Code actually reads. It is regenerated from the two source files by `npm run vscode:sync`. Manual edits to `settings.json` will be overwritten on the next sync and flagged by the audit.

## Merge semantics

The script performs a deep merge:

* Plain objects are merged recursively — keys from `local` are added to or override matching keys from `base`.
* All other values (strings, numbers, booleans, arrays) from `local` replace the `base` value.
* Keys present only in `base` are kept as-is.
* Keys present only in `local` are added to the output.

In practice: `local` wins on every leaf key that appears in both files. Array values are fully replaced, not concatenated.

## npm scripts

| Command | What it does |
| --- | --- |
| `npm run vscode:sync` | Merge base + local and write `settings.json`. Run after editing either source file. |
| `npm run vscode:check` | Exit non-zero if `settings.json` is missing or out of date. Used in CI (`ci:check:vscode`). |
| `npm run vscode:audit` | Full drift report: new keys in output, value mismatches, redundant local keys, local overrides. |
| `npm run build:vscode:recommendations` | Regenerate `.vscode/extensions.json` from installed extensions. |

`postinstall:vscode` runs `vscode:sync` automatically after every `npm install`.

## What belongs where

**Put in `settings.base.jsonc`:**

* Tool configuration paths (`biome.configurationPath`, `vale.valeCLI.config`, `markdownlint.configFile`, `stylelint.configFile`).
* Plugin enable/disable flags (`stylelint.enable`, `css.validate: false`).
* File associations and nesting patterns.
* Editor behaviour that should be consistent across machines (`editor.formatOnSave`, `editor.codeActionsOnSave`).
* Project-wide color theme (`peacock.color`) — the color identifies this workspace.

**Put in `settings.local.jsonc`:**

* Values generated by extensions that write to `settings.json` directly (Peacock's `workbench.colorCustomizations`).
* Anything that varies between machines: absolute paths, proxy settings, account-linked tokens.

**Never edit `settings.json` directly.** VS Code's Settings UI and extensions write to `settings.json`, which is why drift occurs. After VS Code modifies `settings.json`, run `npm run vscode:audit` to see what changed, then carry the new values back into the appropriate source file and re-run `npm run vscode:sync`.

## Audit output

`npm run vscode:audit` exits with code 3 if any issues are found. The report has five sections:

| Section | What it means |
| --- | --- |
| New settings found only in settings.json | Keys that exist in the output but not in base or local. Usually the result of an extension or the VS Code UI writing directly to `settings.json`. Move these to the correct source file. |
| Settings whose effective value differs | The merged value of base + local does not match the current output. Indicates a manual edit to `settings.json`. Correct the source file, then re-sync. |
| Local overrides base | Informational: keys where local supplies a different value than base. Expected and normal. |
| Redundant local settings | Keys in local whose value is identical to base. Remove them from local. |
| Source settings missing from output | Keys in base or local that do not appear in the merge result. Should not happen with a clean merge; investigate if reported. |

## Optimization opportunities

**1. Move Peacock writes to local.**
Peacock writes both `peacock.color` and `workbench.colorCustomizations` to `settings.json` when you pick a color via the command palette. Since `workbench.colorCustomizations` already lives in `settings.local.jsonc`, consider also moving `peacock.color` there so both Peacock-related keys stay together in local.

**2. Add `vscode:audit` to the pre-commit hook.**
Currently `lint-staged` does not run the audit. Adding `npm run vscode:audit` (or at minimum `npm run vscode:check`) to the pre-commit stage via `lint-staged` would catch manual `settings.json` edits before they are committed.

**3. Gitignore `settings.json` and generate it on CI.**
Because `settings.json` is generated, it could be excluded from version control and regenerated on every machine (including CI) via the `postinstall:vscode` hook. The tradeoff is that VS Code will not work in a fresh clone until `npm install` has run.

**4. Add `--verbose` flag to `vscode:sync` in development.**
Running `npm run vscode:sync -- --verbose` shows which files were read and written. Useful when debugging merge issues.

## Script internals

`src/scripts/vscode/merge-vscode-config.ts` is a self-contained TypeScript script with no Astro or project dependencies beyond `jsonc-parser`. It can be run with Node.js 22+ native TypeScript stripping (`node`) or with `npx tsx`.

Key implementation details:

* `readJsoncObject` uses `jsonc-parser` to support trailing commas and `//` comments in the source files.
* `deepMerge` is non-destructive: it creates a new object at every level rather than mutating the inputs.
* The `--check` mode compares the file on disk against the in-memory merge result string-for-string, so whitespace differences count as drift.
* `flattenObject` produces dot-notation paths (`editor.codeActionsOnSave.source.fixAll.biome`) used for the audit diff. Note: array values are represented as leaf nodes (the whole array is a single entry), not flattened further.
* Exit code 3 from `--audit` is distinct from code 1 (argument/runtime error) and code 2 (missing required file), which makes it usable in scripts that want to distinguish "audit found issues" from "script crashed".
